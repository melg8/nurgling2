## ADDED Requirements

### Requirement: Обнаружение перехода между слоями
Система ДОЛЖНА обнаруживать переход игрока между слоями карты через портал (cave, minehole, ladder) и отличать его от других видов телепортации (hearthfire, totem, signpost). Переходы через cellar НЕ размечаются.

#### Scenario: Игрок входит в cave на поверхности
- **WHEN** игрок кликает на портал cave на поверхности (сегмент 1)
- **THEN** система обнаруживает переход gridId → новый gridId и определяет это как переход через портал, а не телепорт

#### Scenario: Игрок телепортируется на hearthfire
- **WHEN** игрок использует телепортацию на hearthfire
- **THEN** система НЕ определяет это как переход через портал и не создаёт маркеры

#### Scenario: Игрок проходит через minehole между уровнями подземелья
- **WHEN** игрок кликает на minehole на уровне N подземелья
- **THEN** система обнаруживает переход между уровнями подземелья (сегмент ≠ 1 → сегмент ≠ 1)

---

### Requirement: Генерация уникального идентификатора (UID)
Система ДОЛЖНА генерировать уникальный 6-символьный alphanumeric идентификатор (0-9a-zA-Z) для каждой пары связанных маркеров. UID ДОЛЖЕН быть одинаковым при движении через портал в обе стороны.

#### Scenario: Генерация UID для cave
- **WHEN** игрок переходит через cave между поверхностью (сегмент 1) и подземельем (сегмент 12345)
- **THEN** система генерирует UID = Base62(SHA256(XOR(1,12345) : coordX : coordY))[0:6], например "ABC123"

#### Scenario: Одинаковый UID при обратном переходе
- **WHEN** игрок переходит через тот же портал в обратном направлении (подземелье → поверхность)
- **THEN** система генерирует тот же UID "ABC123" (благодаря коммутативности XOR: 1^12345 = 12345^1)

#### Scenario: Разные UID для разных порталов
- **WHEN** существуют два разных cave между одними и теми же сегментами
- **THEN** каждый портал получает уникальный UID благодаря разным координатам

---

### Requirement: Создание парных маркеров IN/OUT
Система ДОЛЖНА создавать два связанных маркера с одинаковым UID и суффиксами IN/OUT для указания направления движения.

#### Scenario: Создание маркеров для cave
- **WHEN** игрок переходит через cave с поверхности на подземелье
- **THEN** создаётся маркер "Cave ABC123 IN" на поверхности и "Cave ABC123 OUT" на подземелье

#### Scenario: Создание маркеров для minehole/ladder
- **WHEN** игрок переходит через minehole с уровня N на уровень N+1
- **THEN** создаётся маркер "Hole ABC123 IN" на уровне N и "Hole ABC123 OUT" на уровне N+1

---

### Requirement: Определение направления (IN/OUT)
Система ДОЛЖНА определять направление движения через портал на основе типа портала и сегментов перехода.

#### Scenario: Cave с поверхности на подземелье
- **WHEN** переход из сегмента 1 (поверхность) в сегмент ≠ 1 (подземелье) через cave
- **THEN** направление = IN (вход в пещеру)

#### Scenario: Cave из подземелья на поверхность
- **WHEN** переход из сегмента ≠ 1 (подземелье) в сегмент 1 (поверхность) через cave
- **THEN** направление = OUT (выход из пещеры)

#### Scenario: Minehole (вниз)
- **WHEN** игрок переходит через minehole между уровнями подземелья
- **THEN** направление = IN (всегда вниз, на более глубокий уровень)

#### Scenario: Ladder (вверх)
- **WHEN** игрок переходит через ladder между уровнями подземелья
- **THEN** направление = OUT (всегда вверх, на более мелкий уровень)

#### Scenario: Cellar — исключение из разметки
- **WHEN** игрок переходит через cellar (поверхность → подвал)
- **THEN** маркеры НЕ создаются, поскольку карта cellar всегда небольшой прямоугольник и не требует навигационной разметки

---

### Requirement: Дедупликация маркеров
Система ДОЛЖНА предотвращать создание дубликатов маркеров при повторном проходе через тот же портал.

#### Scenario: Повторный проход через тот же портал
- **WHEN** игрок проходит через портал с UID "ABC123", для которого уже существуют маркеры
- **THEN** система НЕ создаёт новые маркеры, а использует существующие

#### Scenario: Проверка существующих маркеров
- **WHEN** система ищет существующие маркеры с UID "ABC123" на целевом сегменте
- **THEN** если маркеры найдены, создание пропускается

---

### Requirement: Логирование событий
Система ДОЛЖНА логировать все события перехода через порталы и создания маркеров для отладки.

#### Scenario: Логирование перехода
- **WHEN** происходит переход между слоями
- **THEN** в logs/portal_transitions.log записывается: fromSegment, toSegment, portalCoordinates, portalName, timestamp

#### Scenario: Логирование создания маркера
- **WHEN** создаётся новый маркер
- **THEN** в logs/marker_events.log записывается: uid, direction, segmentId, coords, markerName

#### Scenario: Логирование генерации UID
- **WHEN** генерируется UID
- **THEN** в logs/uid_generation.log записывается: input (segmentPair:coordX:coordY), output (UID)

---

### Requirement: Независимое обнаружение переходов
Система ДОЛЖНА иметь собственный независимый трекер для обнаружения переходов между слоями, не полагаясь на PortalTraversalTracker или другие внешние системы навигации.

#### Scenario: Работа при отключённом ChunkNav
- **WHEN** ChunkNav overlay отключён (config.chunkNavOverlay = false)
- **AND** portalMarkerAutoCreate = true
- **THEN** система обнаруживает переходы и создаёт маркеры независимо от ChunkNav

#### Scenario: Обнаружение через изменение gridId
- **WHEN** игрок переходит через портал
- **THEN** система обнаруживает изменение gridId через MCache.getgridt() и определяет это как переход

#### Scenario: Игнорирование телепортов
- **WHEN** игрок использует hearthfire, totem или signpost
- **THEN** система НЕ создаёт маркеры, поскольку это не переход между слоями

#### Scenario: Отключение функции
- **WHEN** конфигурация portalMarkerAutoCreate = false
- **THEN** система НЕ обнаруживает переходы и не создаёт маркеры

---

### Requirement: Массовая разметка порталов на той же карте
При переходе через портал система ДОЛЖНА размечать не только тот портал, через который прошёл игрок, но и все другие порталы на той же карте (сегменте), которые требуют разметки и ещё не имеют связанных маркеров.

#### Scenario: Разметка всех cave на поверхности при переходе через одну
- **WHEN** игрок переходит через cave_1 между поверхностью (сегмент 1) и подземельем (сегмент 12345)
- **AND** на поверхности находятся другие cave (cave_2, cave_3), которые ещё не размечены
- **THEN** создаются связанные маркеры для всех cave: cave_1, cave_2, cave_3 с уникальными UID для каждого

#### Scenario: Пропуск уже размеченных порталов
- **WHEN** на той же карте есть портал cave_2 с уже созданными связанными маркерами (UID "XYZ789")
- **THEN** система НЕ создаёт новые маркеры для cave_2, а переходит к следующему неразмеченному порталу

#### Scenario: Разметка всех minehole на уровне подземелья
- **WHEN** игрок переходит через minehole_1 между уровнем 1 (сегмент 100) и уровнем 2 (сегмент 200)
- **AND** на уровне 1 находятся другие minehole (minehole_2, minehole_3), которые ещё не размечены
- **THEN** создаются связанные маркеры для всех minehole: minehole_1, minehole_2, minehole_3 с уникальными UID
