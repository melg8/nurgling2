## Context

**Проблема:** При переходе через порталы между слоями карты (cave, minehole, ladder) игроки не видят связи между маркерами на разных уровнях. Одинаковые маркеры на поверхности и под землёй не выглядят как связанные переходы.

**Текущее состояние:**
- Маркеры создаются вручную игроком или автоматически для cave burrow
- PortalTraversalTracker существует для навигации, но может быть отключён
- Нет автоматического связывания маркеров между слоями

**Ограничения:**
- Cellar исключён из разметки (карта всегда небольшой прямоугольник)
- Функция должна быть отключаемой через конфиг `portalMarkerAutoCreate`
- **Система должна быть полностью независимой от PortalTraversalTracker**
- Обнаружение переходов через MCache (gridId) и MapFile (segmentId)

**Заинтересованные стороны:**
- Игроки — улучшенная навигация между слоями
- Разработчики — модульная архитектура, логирование для отладки

---

## Goals / Non-Goals

**Goals:**
- Автоматическое создание связанных маркеров при переходе через портал
- Уникальный 6-символьный UID для идентификации пары маркеров
- Суффиксы IN/OUT для указания направления
- Поддержка cave (поверхность ↔ подземелье) и minehole/ladder (между уровнями)
- Массовая разметка всех порталов на сегменте при переходе через один
- Дедупликация — не создавать дубликаты для уже размеченных порталов
- Логирование событий для отладки
- **Полная независимость от PortalTraversalTracker и ChunkNav**

**Non-Goals:**
- Разметка cellar (исключено из требований)
- Изменение существующей логики `PortalTraversalTracker` для навигации
- Поддержка пользовательских типов порталов
- Синхронизация маркеров между клиентами (локальные маркеры)

---

## Decisions

### 1. Архитектура: полностью независимый PortalMarkerTracker

**Решение:** Создать отдельный независимый сервис `PortalMarkerTracker`, который самостоятельно обнаруживает переходы и вызывает `PortalMarkerLinker` для создания маркеров. Система НЕ зависит от `PortalTraversalTracker` или ChunkNav.

**Компоненты:**
```
┌─────────────────────────────────────────┐
│  PortalMarkerTracker (независимый)     │
│  ├─ Мониторит gridId через MCache      │
│  ├─ Мониторит segmentId через MapFile  │
│  ├─ Кэширует портал из lastActions     │
│  └─ Вызывает PortalMarkerLinker        │
└─────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│  PortalMarkerLinker                     │
│  ├─ Генерирует UID                     │
│  ├─ Создаёт маркеры IN/OUT             │
│  ├─ Дедупликация                       │
│  └─ Логирование                        │
└─────────────────────────────────────────┘
```

**Альтернативы:**
- **Интеграция с PortalTraversalTracker** — отклонено: система может быть отключена, не должна зависеть от ChunkNav
- **Использование существующего трекинга навигации** — отклонено: нарушает независимость, усложняет поддержку

**Преимущества:**
- Полная независимость от ChunkNav/PortalTraversalTracker
- Работает даже когда навигация отключена
- Проще тестировать и отлаживать
- Явная ответственность: только маркеры, только переходы

---

### 2. Генерация UID: XOR сегментов + SHA-256 + Base62

**Решение:** 
```
segmentPair = fromSegmentId ^ toSegmentId  // XOR для коммутативности
input = "segmentPair:coordX:coordY"
hash = SHA256(input)
uid = Base62(hash)[0:6]  // 6 символов: 0-9a-zA-Z
```

**Альтернативы:**
- **Хэш только от координат** — отклонено: разные порталы между теми же сегментами получат одинаковый UID
- **Случайная генерация** — отклонено: невозможно воспроизвести тот же UID при обратном переходе
- **MD5 вместо SHA-256** — отклонено: SHA-256 надёжнее, производительность не критична

**Преимущества:**
- XOR коммутативен: `1 ^ 12345 = 12345 ^ 1` → одинаковый UID в обе стороны
- Координаты различают порталы между одними сегментами
- Base62 даёт удобочитаемые UID (без специальных символов)

---

### 3. Направление IN/OUT: на основе типа портала и сегментов

**Решение:**
| Портал | Переход | Направление |
|--------|---------|-------------|
| Cave | Surface (1) → Underground (≠1) | IN |
| Cave | Underground (≠1) → Surface (1) | OUT |
| Minehole | Level N → Level N+1 | IN (всегда вниз) |
| Ladder | Level N+1 → Level N | OUT (всегда вверх) |

**Альтернативы:**
- **Определять по позиции игрока относительно портала** — отклонено: сложно, ненадёжно
- **Всегда IN для входа в здание/пещеру** — отклонено: не работает для minehole/ladder

**Преимущества:**
- Простая детерминированная логика
- Учитывает семантику порталов (minehole всегда вниз, ladder всегда вверх)

---

### 4. Массовая разметка: сканирование сегмента при переходе

**Решение:** При переходе через портал сканировать все порталы на исходном сегменте и создавать маркеры для всех неразмеченных.

**Алгоритм:**
```
1. Определить сегмент перехода (fromSegmentId)
2. Найти все порталы на сегменте (cave, minehole, ladder)
3. Для каждого портала:
   a. Проверить, есть ли маркеры с таким UID (дедупликация)
   b. Если нет → сгенерировать UID, создать пару IN/OUT
```

**Альтернативы:**
- **Размечать только пройденный портал** — отклонено: требует многократных проходов
- **Предварительное сканирование всех сегментов** — отклонено: избыточно, требует хранения состояния

**Преимущества:**
- Все маркеры создаются за один переход
- Не требует предварительного знания о порталах
- Дедупликация предотвращает повторную обработку

---

### 5. Хранение маркеров: SMarker (системные маркеры)

**Решение:** Использовать `MapFile.SMarker` вместо `PMarker` для связанных маркеров.

**Альтернативы:**
- **PMarker (игроцкие маркеры)** — отклонено: игрок может случайно удалить
- **Виртуальные маркеры в памяти** — отклонено: не сохраняются между сессиями

**Преимущества:**
- SMarker сохраняются в файле карты
- Не отображаются в UI как игроцкие маркеры
- Автоматически появляются во всех контекстах (MiniMap, MapWnd, NMapWnd)

---

### 6. Логирование: текстовые файлы в logs/

**Решение:**
- `logs/portal_transitions.log` — переходы между слоями
- `logs/marker_events.log` — создание/удаление маркеров
- `logs/uid_generation.log` — генерация UID

**Альтернативы:**
- **Java Logger** — отклонено: может не работать в среде игры
- **Консольный вывод** — отклонено: сложно отлаживать постфактум
- **SQLite для событий** — отклонено: избыточная сложность

**Преимущества:**
- Простой формат для чтения
- Легко фильтровать по типу события
- Не требует внешних зависимостей

## Risks / Trade-offs

### [Risk] Конфликты UID при редких коллизиях SHA-256

**Описание:** Теоретически возможны коллизии хэшей при большом количестве порталов.

**Mitigation:**
- SHA-256 имеет 2²⁵⁶ возможных значений → вероятность коллизии ~10⁻⁷⁷
- Base62 сокращает до 62⁶ = 56,8 млрд комбинаций
- При коллизии портал получит некорректный UID → игрок заметит, можно исправить вручную

---

### [Risk] Производительность при массовой разметке

**Описание:** Сканирование всех порталов на сегменте может замедлить переход.

**Mitigation:**
- Ограничить сканирование только порталами (не все gobs)
- Кэшировать результаты проверки дедупликации
- Логировать время выполнения для мониторинга

---

### [Risk] Неправильное определение направления для minehole/ladder

**Описание:** Если minehole/ladder расположены нестандартно, направление может быть неверным.

**Mitigation:**
- Использовать эвристику: если рядом ladder → пришли сверху (OUT), если minehole → пришли снизу (IN)
- Добавить логирование для отладки спорных случаев
- Позволить игроку вручную переименовать маркер при необходимости

---

### [Risk] Потеря маркеров при обновлении карты

**Описание:** При изменении сегментов карты (обновление сервера) маркеры могут потерять привязку.

**Mitigation:**
- UID генерируется от XOR сегментов → при изменении сегментов UID изменится
- Игрок может повторно пройти через портал для создания новых маркеров
- В будущем: сохранение UID в постоянном хранилище

---

### [Trade-off] Простота vs Полнота

**Описание:** Выбран простой подход с базовой функциональностью вместо сложной системы с правилами трансформации координат.

**Обоснование:**
- MVP достаточно для улучшения навигации
- Можно расширить в будущем (преобразование относительных координат)
- Меньше риска ошибок в первой версии

## Migration Plan

**Шаги развёртывания:**

1. **Добавить новые классы:**
   - `PortalMarkerLinker.java` — основной сервис для создания маркеров
   - `PortalMarkerTracker.java` — **независимый** трекер для обнаружения переходов
   - `LayerTransition.java` — модель данных перехода
   - `MarkerUidGenerator.java` — генератор UID
   - `PortalMarkerLink.java` — представление связи
   - `PortalName.java` — утилиты имён и направлений
   - `PortalMarkerLogger.java` — логгер событий

2. **Интеграция в игровой цикл:**
   - Вызов `PortalMarkerTracker.tick()` из `NMapView.tick()` или аналогичного места
   - Альтернатива: вызов из `NCore.tick()` если существует

3. **Добавить конфиг:**
   - `portalMarkerAutoCreate = true` (по умолчанию)
   - Опция в UI (QoL settings) для включения/выключения

4. **Тестирование:**
   - Пройти через cave (поверхность → подземелье)
   - Проверить создание маркеров "Cave XXX IN" и "Cave XXX OUT"
   - Пройти обратно → проверить дедупликацию
   - Пройти через minehole/ladder → проверить "Hole XXX IN/OUT"
   - **Проверить работу при отключённом ChunkNav overlay**

**Откат:**
- Установить `portalMarkerAutoCreate = false`
- Удалить созданные маркеры вручную через UI карты
